# 社区记忆库本地化实装规划报告（Phase 0：不连接服务器）

> 文档日期：2026-03-08（America/New_York）  
> 当前约束：暂不连接任何后端服务；媒体内容全部内置在本地 App。  
> 目标：先把 Active / Passive 双模式在端内跑通，再进入后端化阶段。

## 0. 执行结论

本阶段采用 **Local-First 架构**：

1. 不调用远端 API，不做 APNs，不依赖云存储。
2. 记忆数据和媒体走本地资源（`Bundle` + 本地持久化）。
3. Active 模式与 Passive 模式都在端上闭环完成。
4. “已探索档案”和“记忆卡片”先保存为本地状态与本地文件。

## 0.1 强制实现约束（新增）

1. 新功能实现必须基于当前项目既有设计风格、组件层级和制作规范，不得另起一套视觉与交互语言。
2. 新功能落地阶段禁止直接使用 SwiftUI API 实现 UI 交互与页面装配。
3. UI 层统一走现有规范化组件与当前工程可接受的非 SwiftUI 落地方式（例如现有桥接层、UIKit 能力）。
4. 代码评审新增门禁：若新增功能直接引入 SwiftUI API，则视为不通过。

## 1. 业务链路（本地闭环版）

### 1.1 Active 模式

1. 用户点击地图 `pin`，进入导航态。
2. App 在端上做“到达范围判定”（距离 <= `unlockRadius`，可加停留秒数）。
3. 达标后解锁记忆详情，按本地媒体类型渲染：`image` / `audio` / `video` / `ar(usdz)`。
4. 用户点“体验完成”后写入本地档案为 `已探索`。
5. 端上生成分享卡片图片，支持保存相册和系统分享。

### 1.2 Passive 模式

1. 首次启用“附近记忆提醒”时请求通知权限与定位权限。
2. 用本地记忆点计算“最近监控点集合”，注册设备端地理围栏。
3. 用户靠近时由本地通知触发文案，点击后直达 Active 链路。

## 2. 端内架构设计

```text
[iOS App]
  -> LocalMemoryRepository (本地记忆数据)
  -> LocalMediaCatalog (Bundle 媒体索引)
  -> GeoEngine (距离判定 + geofence 管理)
  -> ExplorationStore (已探索记录)
  -> CardRenderer (端上卡片渲染)
  -> NotificationOrchestrator (本地通知编排)
```

### 2.1 模块职责

1. `LocalMemoryRepository`：加载本地记忆数据（JSON/SQLite/CoreData）。
2. `LocalMediaCatalog`：将 `memoryId -> 本地媒体资源` 建索引。
3. `GeoEngine`：负责最近点计算、进入范围判定、地理围栏轮换。
4. `ExplorationStore`：记录探索状态、完成时间、来源（active/passive）。
5. `CardRenderer`：把记忆内容渲染为图片（卡片模板）。
6. `NotificationOrchestrator`：注册本地提醒、去重与冷却。

## 3. 本地数据模型（建议）

1. `MemoryItem`
`id, title, story, latitude, longitude, unlockRadiusM, tags, media[]`
2. `MemoryMedia`
`id, memoryId, type(image/audio/video/ar), localAssetName, duration, thumbnailAssetName`
3. `ExplorationRecord`
`id, memoryId, exploredAt, source(active/passive), cardLocalPath`
4. `WatchRegion`
`memoryId, center, radiusM, enabledAt, cooldownUntil`

> 建议：`MemoryItem` 与 `MemoryMedia` 放只读本地包；`ExplorationRecord` 与 `WatchRegion` 放可写本地存储（CoreData/SQLite）。

## 4. 状态机与判定规则

`IDLE -> NAVIGATING -> IN_RANGE -> UNLOCKED -> EXPERIENCED -> ARCHIVED`

规则：

1. `ARCHIVED` 幂等，同一记忆重复完成不重复建档。
2. 解锁判定建议“双条件”：`distance <= unlockRadius` + `dwellSeconds >= x`（比如 8-15 秒）。
3. Passive 触发后，若该记忆已 `ARCHIVED`，默认不再重复通知。

## 5. Active 模式实现细节（无服务端）

1. 点击 pin：创建本地 `NavigationSession`（内存态即可）。
2. 导航期间持续取位置，实时算到目标点距离。
3. 达标触发本地解锁事件，详情页从 `LocalMemoryRepository` 读取。
4. 媒体渲染：
`image` 用本地图片资源；`audio/video` 用本地文件 URL；`ar` 用本地 `usdz` + Quick Look。
5. 体验完成后写 `ExplorationRecord`，并触发本地卡片渲染。

## 6. Passive 模式实现细节（无服务端）

### 6.1 权限策略

1. 首屏先解释“附近记忆提醒价值”，再触发权限请求。
2. 权限顺序建议：通知权限 -> 定位权限（含后台用途文案）。

### 6.2 地理围栏策略

1. 设备同时监控数量有上限，需仅保留“最近 N 个”（N 建议 <= 20）。
2. 每次位置显著变化后，重算最近集合并替换监控列表。
3. 对同一记忆设置冷却窗口（例如 24h）避免频繁打扰。

### 6.3 通知触发与跳转

1. 进入围栏后下发本地通知（`UNUserNotificationCenter`）。
2. payload 带 `memoryId`，点击后路由到对应记忆详情。
3. 进入详情后复用 Active 的解锁与归档逻辑。

## 7. 记忆卡片（本地生成）

1. 输入：标题、摘要、地点名、时间、封面图、本地配色模板。
2. 输出：卡片图片文件（App 沙盒）。
3. 分享：
保存相册（`PHPhotoLibrary`）；
系统分享（`UIActivityViewController`）。

## 8. 本地方案边界（已知限制）

1. 无法做远程运营（推送 AB、服务端实时策略）。
2. 设备丢失会丢本地探索档案（除非后续加 iCloud/后端同步）。
3. 地理围栏数量有限，长尾记忆点需动态轮换而非全量常驻。

## 9. 分阶段实施（仅本地）

### Phase 0A（1 周）

1. 落地本地记忆数据模型与媒体索引。
2. 跑通 Active：导航 -> 解锁 -> 详情媒体播放。

### Phase 0B（1 周）

1. 落地探索归档与档案页展示。
2. 落地卡片本地渲染 + 保存相册 + 系统分享。

### Phase 0C（1 周）

1. 落地 Passive：本地 geofence + 本地通知 + 点击跳转。
2. 完成去重/冷却与边界回归测试。

## 10. 本周可直接开工清单

1. 新建 `LocalMemoryRepository`，先从本地 `JSON` 读取记忆与媒体元数据。
2. 定义 `MemoryMedia.type` 与本地资源命名规范（含 `usdz`）。
3. 实现 `UnlockEvaluator`（距离 + 停留时长）。
4. 实现 `ExplorationStore`（CoreData/SQLite 二选一）。
5. 实现 `CardRenderer`（固定模板 1 套即可）。
6. 接入本地通知与地理围栏轮换（最近点 Top-N）。

## 11. 后续迁移到后端（保留位）

当你准备连服务器时，沿用当前端内模型做平滑迁移：

1. `LocalMemoryRepository` 升级为 `Repository` 协议：本地实现 + 远端实现可切换。
2. `ExplorationStore` 新增同步队列，离线写本地、在线回传服务端。
3. 通知从本地触发逐步过渡到“本地 + APNs 混合模式”。
