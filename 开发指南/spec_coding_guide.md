# YOM 项目 Spec Coding 实操指南

这份指南将手把手教你如何在 YOM项目中实施 Spec Coding。

## 准备阶段：明确你的新角色
在 Spec Coding 模式下，你需要完成一次思维视角的转换：
**你不再是一个绞尽脑汁寻找 Bug 的“码农”，而是一个发号施令的“产品经理（PM） / 架构师 / 测试工程师（QA）”。**
这三位一体的角色决定了你与 AI 交互的方式。

---

## 核心工作流：四步法

在开发任何一个新功能（Feature）时，请严格遵循以下四个步骤。我们以“在 YOM 中添加一个用户设置页面”为例：

### 第一步：编写 Spec (规格文档)

这是最重要的一步，也是你花费时间最多的一步。高质量的 Spec 决定了 AI 生成代码的上限。

1. **新建文件**：在 `specs/` 目录下创建一个新的 Markdown 文件，例如：`specs/001-settings-page.md`。
2. **结构化你的需求**：好的 Spec 需要包含以下核心要素：
   * **功能目标（Goal）**：一句话概括这个页面的目的。
   * **依赖关系（Context/Dependencies）**：告诉 AI 需要参考哪些现有文件或库（例如：必须使用设计规范 `design/ios_ui_layout_guidelines.md`，或依赖 `Shared/Models/User.swift`）。**精准提供上下文是避免 AI 产生幻觉的关键。**
   * **UI 规格（UI Specification）**：
     * 需要哪些控件（如：Toggle, NavigationLink, Text）。
     * 明确的视觉要求（颜色、间距、字体），不要模糊地说“好看一点”，而是说“背景使用纯白 #FFFFFF，标题使用 Large Title 34px Bold”。
   * **行为/业务逻辑（Behavior/Business Logic）**：
     * 当用户点击按钮时，状态应当如何改变。
     * 数据如何持久化（例如：“使用 `@AppStorage` 保存夜间模式状态”）。
   * **验收标准（Acceptance Criteria）**：
     * 这是 QA 的职责。列出 3-5 条明确的、可测试的条件（例如：“当开启夜间模式 Toggle 时，全局主题必须立即切换到 Dark 模式”）。

### 第二步：编写测试 (TDD - Test Driven Development)

在让 AI 写业务代码之前，先让它写测试。这听起来反直觉，但在 Spec Coding 中这是确保代码质量的核武器。

1. **提交 Spec 给 AI**：“这是 `specs/001-settings-page.md`，请**只根据这个文档**，在 `YOMUITests/SettingsPageTests.swift` 中为我编写 UI 测试用例。**请不要在 Features 目录下写任何业务代码。**”
2. **运行测试（必然失败）**：此时你的业务代码还不存在，所以测试会编译失败或运行失败。这属于预期的 **Red**（红色）阶段。

### 第三步：生成与验证 (Generate & Verify)

现在，让 AI 登场完成“码农”的工作。

1. **下达生成指令**：“现在，测试已经存在。请根据 `specs/001-settings-page.md` 的描述，在 `Features/Settings/` 目录下完整实现 `SettingsView.swift` 和对应的 ViewModel。**目标：确保刚才你写的测试能够全部通过。**”
2. **运行测试并验证**：构建项目，运行测试。
   * 如果通过（**Green**）：恭喜，这部分核心功能已经完成。
   * 如果失败：**克制住手动修改代码的冲动！** 把错误日志发给 AI：“测试失败了，报错是...，请修复你的代码以通过测试。”
   * 如果依然反复失败或逻辑不符合预期：这通常意味着你的 **Spec 写得不够清楚或有歧义**。
3. **退回第一步修改 Spec**：不要让 AI “看着办修改”。你应该去修改 `specs/001-settings-page.md`，补充遗漏的逻辑或澄清模糊的表述，然后让 AI **重新基于新的 Spec 全盘生成**。

### 第四步：沉淀与重构 (Refactoring & Rules)

通过了一次开发周期后，你需要沉淀经验。

1. **提取通用规则**：在刚才的开发中，如果你发现 AI 多次使用了错误的颜色或者冗余的修饰符，你需要将正确的约束写入根目录的 `CLAUDE.md`（或全局 Prompt）中。
   * 例：在 `CLAUDE.md` 中补充：“*UI 规范：所有列表页禁止使用系统自带的 List separators，必须使用自定义的分隔线。*”
2. **更新项目配置文件**：因为你是落盘新文件（如 `SettingsView.swift`），所以你需要检查并确保在 `project.yml` 中配置了正确的 source 路径（现在你的 YOM 配置是只要在 `Features` 目录下就会自动包含，所以这步通常是自动的）。
3. **运行 `xcodegen`**：在终端运行编译命令（如 `xcodegen generate`，如果这是你的流程）来重新生成 `.xcodeproj`，然后编译运行最终确认。

## 极简流程速记（每次新增功能前默念）

1. **写档**：在 `specs/` 下写清 PM、UI、QA 的所有要求，并链接好上下文。
2. **写测**：让 AI 先把验收标准翻译成测试代码跑起来（红灯）。
3. **写码**：让 AI 照着 Spec 写业务代码，直到测试跑通（绿灯）。
4. **遇错**：绝不手撸，只改 Spec / Prompt，让 AI 重写。
5. **落规**：把总结的坏味道和架构铁律补充进 `CLAUDE.md`。
