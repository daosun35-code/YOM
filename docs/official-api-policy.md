# Official API Policy

> 本文件定义技术选型的 API 优先级阶梯与例外审批规则。

## 升级阶梯（按优先级递降）

| 阶梯 | 允许条件 | 示例 |
|------|---------|------|
| **L1 — SwiftUI 原生** | 默认首选，无需审批 | `TabView`, `NavigationStack`, `.sheet`, `.fullScreenCover`, `.inspector`, `.searchable` |
| **L2 — SwiftUI + 系统行为接受** | 官方 API 可实现 ≥ 80% 目标时，接受系统默认外观与行为，不做自定义替代 | 系统标准 Tab Bar 样式、系统搜索栏交互 |
| **L3 — UIKit 官方桥接** | 仅当 SwiftUI 无法完成关键用户流程时启用；须在代码注释中标注桥接理由 | `UIViewControllerRepresentable` 包装 `MKMapView` |
| **L4 — 自定义容器/动画** | 必须有 ADR 审批记录 (`docs/adr/`) | 自定义转场动画、自定义容器视图 |

## 执行规则

1. **逐级递降**：每个 UI 需求从 L1 开始评估，仅当当前阶梯无法满足时才降至下一级
2. **视觉优化不阻塞功能**：纯视觉打磨（如微调阴影、渐变）不得成为功能上线的阻塞项
3. **80% 规则**：若 L1 已能覆盖目标体验的 80%，直接采用系统行为；剩余 20% 作为后续迭代项而非首轮自定义借口
4. **桥接声明**：每处 UIKit 桥接须在对应源文件顶部注释中注明理由与预期的官方 API 替代路径
5. **自定义禁止回流**：不得以"还原旧版视觉效果"为由申请 L4

## ADR 审批要求

- 任何 L4 级选型必须提交 ADR 草稿至 `docs/adr/NNN-<title>.md`
- ADR 格式最低要求：标题、背景、决策、后果、审批人
- ADR 必须由人工审批——AI 可协助起草但不得自行批准
- **Beta 期间时效约束**：因"官方 API 暂不支持"而批准的例外，每次 Beta 更新后须重新评估；官方已补齐功能时应撤销 ADR 并迁回官方方案

## 审计方式

- 代码审查时可要求作者标注每个 UI 组件所属阶梯
- `specs/` 中的验收场景须标注预期使用的 API 阶梯
- 定期扫描项目中 L3/L4 组件数量，持续推动向 L1/L2 迁移
